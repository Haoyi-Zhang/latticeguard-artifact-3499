# LatticeGuard Artifact (Anonymous Repository Packet)

This artifact accompanies the paper "LatticeGuard: Law-Level Metamorphic Regression Oracles for Access-Control Policy Evaluators." It is a deterministic, CPU-native, offline replay package for law-level metamorphic testing of access-control policy evaluators.

## Scope

The counted run executes three real local adapters: `casbin_py` via PyCasbin, `cedar_py` via cedarpy, and `opa_rego_cli` via a shipped OPA/Rego CLI executable admitted by the SHA-256 pinning/preflight contract under `tools/OPA_PINNING.md`. If the OPA executable is removed or hash-mismatched, it is excluded before any OPA result is observed and the denominator is recomputed.

The repository is designed so the core research object is not hidden in prose. Applicability predicates, invalid-transformation rejection, reference soundness, theorem/proof-obligation ledgers, bounded core model checking, native benchmark import, counterexample replay, coverage, source provenance, and claim binding are all first-class ledgers with dedicated verifiers.

Scope guardrails:

- The default counted run validates exactly three real adapters, `casbin_py`, `cedar_py`, and hash-gated `opa_rego_cli`.
- OPA/Rego rows enter the denominator only after the shipped executable passes the pinned SHA-256 contract; otherwise OPA is recorded as a pre-result exclusion.
- The 5,040 minimized counterexamples are seeded synthetic controls for replay and minimization, not observed production defects.

## Adapter, corpus, and claim boundaries

The OPA/Rego adapter executes OPA as an external CLI, but it uses the repository's normalized law object rather than a native OPA RBAC policy hierarchy. To avoid Rego recursion problems on the locked hierarchy stress cases, `scripts/run_full_evaluation.py` precomputes each principal's transitive role closure with `role_closure()` and supplies the sorted result as `data.reachable_roles`; the Rego rule then checks membership in that relation. This makes OPA a real counted allow/deny evaluator for the normalized fragment, while placing the closure computation in the artifact's trusted semantic core. The closure algorithm is covered by the unit-test replay and by the independent finite law-kernel closure in `latticeguard/law_algebra.py`.

Adapter/reference agreement must be read as an implementation-consistency check. A disagreement would indicate an adapter, normalization, or fixture-generation problem. Agreement across Casbin, Cedar, OPA, and the normalized reference evaluator does not by itself validate the relation catalog; that validation comes from the executable predicates, invalid-transformation rejection rows, reference-soundness rows, theorem/proof-obligation ledgers, bounded finite model checking, and mechanized law-kernel replay.

The 120 source IDs are a frozen stratified corpus, not a claim that every row is an independent real-world benchmark. They consist of 6 documentation-derived slices, 7 upstream example slices, 10 native canonical fixture slices, 96 deterministic semantic stress witnesses, and 1 generated canonical law probe. The stress witnesses are generated from the frozen relation catalog and then kept as deterministic subjects. They are not added or removed after observing Casbin, Cedar, or OPA outcomes, and rejected or unsupported candidates remain visible in the ledgers instead of being hidden to improve the pass rate.

The reported run observed no pre-existing implementation defects in the three counted adapters. The seeded-mutant rows and minimized counterexamples are ground-truth controls for replaying known semantic drift, not organic bug discoveries. The artifact therefore supports claims about reproducible law-level oracle construction, denominator discipline, regression sensitivity, and replay integrity; it should not be described as a bug-finding win over Casbin, Cedar, or OPA.

## System requirements

- Operating system: Linux, tested on the cloud release host.
- Architecture: x86_64/amd64. The bundled Cedar bridge is a CPython 3.13 native extension.
- Python: `python3.13` is required for the default replay. Rebuilding the same pinned `cedarpy` version for another interpreter is out of scope for the clean packet.
- Execution mode: CPU only; no GPU, cloud account, commercial API, private data, or network scanning is required.
- Memory and disk: the current full replay is designed for ordinary reviewer workstations; reserve at least 4 GB RAM and 2 GB free disk for generated results and manifests.
- Unsupported in this packet: ARM, macOS, Windows, and Python versions below 3.13 have not been validated.
- Dependencies: vendored packages live under `vendor_python/`; run commands from `artifact/` so the scripts can prepend the project and vendor paths.

## Evidence reading guide

The reviewer-facing controls should be read as negative or isolation baselines, not as substitutes for the full law-level oracle or the main counted result rows. The rows marked `RANDOM_VALID_PERTURB`, `PROPERTY_GENERATOR_NO_REJECTION`, `CROSS_ADAPTER_DIFFERENTIAL_ONLY`, and `SINGLE_RELATION` are controls; the remaining rows in `baseline_results.csv` are the primary technique evidence.

- `RANDOM_VALID_PERTURB`: budget-matched valid syntax perturbations. This is a noise-floor control; it shows that mere validity and row volume do not imply a counted metamorphic pass/fail.
- `PROPERTY_GENERATOR_NO_REJECTION`: property-style generation with rejection discipline disabled. This is a denominator-safety control; it shows why applicability gating is required.
- `CROSS_ADAPTER_DIFFERENTIAL_ONLY`: normalized adapter comparison only. This is a comparability sanity check, not law-level coverage.
- `SINGLE_RELATION`: one-row-per-relation isolation control. This shows that relation families are not interchangeable and that no single family covers the whole semantics space.

The failure ledgers should be read together:

- `rejections.csv` records 3,240 candidates rejected before counting because the predicate would be unsound if accepted.
- `counterexamples.json` records 5,040 minimized failures with exact before/after policy material, observed decisions, and replayable witnesses; this is a semantic-failure corpus, not the same scope as the bounded model-check ledger.
- `semantic_counterexample_replay.csv` replays the same 5,040 witnesses and passes all of them, which supports replay integrity but does not claim adapter correctness by itself.
- `error_analysis.csv` separates real adapter-law violations from seeded positive controls; the real-adapter row currently records zero observed real-law violations.

The relation-coverage ledger should be read as bounded exact evidence:

- `model_check_summary.json` reports 74,024 bounded cases, 0 failures, 6,336 enumerated policies, and coverage of 12 relation families. The bound is the finite enumeration over the locked schema/policy-search envelope used by the checker, not an open-ended search.
- This supports internal consistency of the frozen relation catalog under the bounded finite model, but it is not an unbounded completeness theorem.

The scalability ledger is engineering-boundary evidence only:

- `scalability.csv` spans rule count, hierarchy depth, hierarchy branching factor, principal/resource/action count, relation families enabled, and adapters enabled.
- The `runtime_ms` and `memory_kb` fields are normalized trend fields and should not be used to claim a performance advantage.
